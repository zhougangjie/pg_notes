# 从函数的角度理解数据库执行器

## 1. 定义阶段：方法签名 (The Signature)

- **SQL 语句** = `Function Definition`
  - **输入 (`FROM`)**：数据源的 Schema（类型定义）。
  - **输出 (`SELECT`)**：结果集的 Schema（返回类型）。
  - **逻辑 (`WHERE/JOIN`)**：函数的约束与处理逻辑。
- **核心特征**：**声明式**。只定义“做什么”（Signature），不定义“怎么做”（Implementation）。

## 2. 编译阶段：验证与代码生成 (Compilation)

- **Parser & Analyzer (编译器前端)**：
  - **动作**：语法检查、语义校验、权限验证。
  - **产出**：**查询树 (Query Tree)** —— 合法的、未优化的逻辑 AST。
- **Optimizer (优化器/代码生成器)**：
  - **动作**：基于成本模型 (CBO)，从成千上万种实现路径中选出最优解。
  - **产出**：**物理计划树 (PlannedStmt)** —— 针对当前数据和硬件优化的“机器码”。

## 3. 运行阶段：实例化与执行 (Runtime)

- **Step 1: 初始化 (`ExecutorStart`) —— 栈帧分配**
  - **动作**：为这次“函数调用”创建运行时上下文 (`EState` + `PlanState` 树)。
  - **细节**：打开表文件（建立参数连接），分配局部变量空间 (`TupleTableSlots`)，预编译表达式。
  - **类比**：函数进栈，提升栈指针，准备局部变量区。

- **Step 2: 执行 (`ExecutorRun`) —— 惰性求值循环**
  - **模式**：**Pull Model (拉模型) / Lazy Evaluation (惰性求值)**。
  - **机制**：
    - 上层节点调用 `ExecProcNode()`（类似 `next()` 或 `yield`）。
    - 下层节点恢复上次暂停的状态，计算**一行**数据，填入 `TupleTableSlot`（返回值），然后再次暂停。
    - **流式处理**：内存恒定，首行极速。
    - **批量/物化**：某些算子（Sort/Hash）需等待所有输入（类似返回 Array），是并发优化的切入点。
  - **类比**：协程 (Coroutine) 或生成器 (Generator) 的反复*resume*。

- **Step 3: 清理 (`ExecutorEnd/Finish`) —— 出栈与副作用**
  - **动作**：销毁栈帧，释放资源。
  - **细节**：关闭文件，重置内存上下文，执行排队的 AFTER 触发器（处理函数副作用）。
  - **类比**：函数返回，弹出栈帧，执行 `finally` 块。

## 模型价值

1.  **解释了性能瓶颈**：
    - **签名复杂** ➡️ 逻辑太重。
    - **编译错误** ➡️ 优化器选错了实现计划（Cost 估算偏差）。
    - **运行低效** ➡️ 内存不足导致频繁磁盘溢出（Swap），或锁竞争。

2.  **解释了架构扩展**：
    - 只要新组件（如列存引擎、FDW）能实现同样的“读取接口”（Signature），就可以无缝替换底层实现，上层 SQL 无需修改。

3.  **解释了并发与一致性**：
    - **Portal** 是函数的**迭代控制器**，支持暂停/恢复。
    - **MVCC** 确保每次“函数调用”看到的“输入参数快照”是一致的，互不干扰。
