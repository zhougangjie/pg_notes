# 提交事务

## 1. CommitTransaction() 的整体流程

提交流程遵循 **“先记账，后确认，再清场”** 的逻辑：

1. **预提交 (Pre-Commit)**：触发所有 `BEFORE COMMIT` 触发器，处理延迟约束检查。
2. **写日志 (WAL Write)**：将事务产生的变更记录（WAL Record）写入日志缓存。
3. **刷盘 (WAL Flush)**：强制将 WAL 缓存持久化到磁盘。
4. **标记提交 (CLOG)**：在提交日志中将 XID 状态改为 `COMMITTED`（这是事务真正生效的原子时刻）。
5. **资源释放 (Cleanup)**：释放锁、删除临时文件、关闭文件描述符、清理内存上下文。

## 2. WAL 日志的作用、写入与刷盘

- **作用**：实现 **D (Durability)**。由于随机写数据页（Data Page）很慢，先顺序写 WAL 日志来保证即使崩溃也能恢复数据。
- **写入 (Write)**：在 `INSERT/UPDATE` 执行过程中，变更先写到内存中的 **WAL Buffer**。
- **刷盘 (Flush)**：在 `COMMIT` 触发时（或 Buffer 满了、或后台进程 Checkpoint 时），调用驱动将日志同步到物理磁盘。

## 3. XLogFlush() 的作用与等待

- **作用**：发出 `fsync()` 指令，确保操作系统的 IO 缓存数据真正落入磁盘介质。
- **为什么等待**：数据库必须保证：**只要告诉用户“提交成功”，数据就绝对不能丢失。** 如果不等待 `XLogFlush` 完成就返回，一旦此时停电，内存中未刷盘的日志丢失，重启后该事务会像从未发生过一样（违背持久性）。

## 4. TransactionIdCommitTree() 的作用

- **做了什么**：在 **CLOG (Commit Log)** 位图中，找到当前事务 ID（以及所有子事务 ID）对应的比特位，将其状态从 `01` (In Progress) 修改为 `10` (Committed)。
- **关键性**：这是 **原子性 (Atomicity)** 的体现。在这一步之前，即便数据已经在磁盘上，其他事务通过 MVCC 检查也会认为你“不可见”；一旦这一步完成，你的修改瞬间对全世界可见。

## 5. 资源清理：内存、锁、文件描述符

- **内存**：销毁 `TransactionContext`。这是事务级的内存池，所有在这个事务中申请的临时内存（如查询计划、中间计算结果）会瞬间成块释放。
- **锁**：调用 `LockReleaseAll`。根据 **VXID** 释放当前进程持有的所有排他锁（如 RowExclusiveLock），允许其他阻塞的事务继续。
- **文件描述符**：关闭事务中打开的临时文件句柄（VFDs），确保资源不泄露。

## 6. synchronous_commit 的影响

这个参数决定了用户**什么时候收到“OK”响应**：

- **on (默认)**：流程走到第 3 步 `XLogFlush` 成功后才返回。最安全，性能稍低。
- **off (异步提交)**：流程走到第 2 步（日志进 Buffer）就直接返回 OK。性能极高（减少 IO 等待），但主机掉电可能导致最近 0.5 秒左右的数据丢失（虽然数据库本身不会损坏）。
- **remote_apply/write**：在物理复制环境下，要求日志不仅刷到本地，还要刷到从库才返回。

## 总结

`CommitTransaction` 的设计核心就是 **“宁可慢，不可丢”**。通过 `XLogFlush` 保证数据在磁盘，通过 `CLOG` 保证状态原子可见。
